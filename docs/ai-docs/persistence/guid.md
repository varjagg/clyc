# GUID

A GUID is the **stable cross-image identity** of a Cyc object. Every constant has one. NARTs do not have their own GUIDs (they're identified by their formula, which resolves to constant GUIDs); variables do not (they have local-image ids only). Inside a single running image, objects are referenced by handle id (an integer), but as soon as a value crosses an image boundary — written to a `.cfasl` file, sent over the API, externalized for transport — the GUID is what re-identifies it on the other side.

The implementation is `guid.lisp` (the type, construction, comparison, byte-vector codec) and a thin layer in `constants-high.lisp` / `constants-low.lisp` (the constant→GUID and GUID→constant tables).

## Why a GUID and not a handle id

Handle ids are dense small integers minted in allocation order. They have two properties that make them useless for cross-image identity:

1. **They depend on the order objects were created in this image.** Two images that loaded the same dump in the same order *would* agree on ids — but two images created independently won't, and a re-dump in a different order changes them.
2. **They're recyclable.** Delete a constant, its id can be reused by the next freshly-minted constant. The handle namespace is per-image.

GUIDs solve this by being:

1. **Stable** — once a constant has a GUID, that GUID stays with it for life. No reassignment.
2. **Globally unique** — generated as 128-bit random (UUID v4), so two independently-created constants cannot collide.
3. **Recoverable** — the GUID-to-constant table is rebuilt on every load, so a serialized GUID can be resolved back to whatever constant currently exists with that GUID.

The cost is 16 bytes vs ~3 bytes (typical handle id). On the wire, CFASL has two formats: opcode 30 (`constant`) writes just the dump-id when both sides share an id space (in-image), opcode 32 (`complete-constant`) writes GUID + name when they don't. See `cfasl.md` for the wire details.

## The `guid` type

A `guid` is a single-slot defstruct wrapping a 36-character UUID string in canonical form (`xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`). The constructor is `%make-guid`; the public-facing entry points are:

| Function | Purpose |
|---|---|
| `new-guid` | Generate a fresh random UUID v4. Random bytes for 16 bytes, then set version (high nibble of byte 6 to 4) and variant (high two bits of byte 8 to `10` per RFC 4122). |
| `string-to-guid string` | Wrap a UUID string in a `guid` struct. No validation. |
| `guid-to-string guid` | Unwrap. |
| `guid-string-p object` | Looser predicate: object is a 36-char string with `-` at positions 8/13/18/23 and only hex/dash elsewhere. |
| `guid-p` | Auto-generated by defstruct. |
| `guid=`, `guid/=`, `guid<`, `guid<=`, `guid>`, `guid>=` | String comparison on the UUID string. `guid=` uses `string-equal` (case-insensitive). |
| `disassemble-guid-to-fixnums guid &optional bv` | Convert to a 16-byte vector. Used by the CFASL terse-encoding path. |
| `assemble-fixnums-to-guid byte-vector` | Inverse. |
| `*dtp-guid*` | Type tag = 127. |

The `print-object` method prints `#G"<uuid>"`.

The `crack-date-relative-guid` / `date-relative-guid-p` / `make-date-relative-guid` family — for GUIDs derived from a build date plus an offset — is unimplemented in the Java and the port. Cyc the engine has them; the LarKC port doesn't.

## Constant ↔ GUID tables

`constants-low.lisp` owns three globals that maintain the mapping:

| Variable | Type | Purpose |
|---|---|---|
| `*constant-guid-table*` | `id-index` (id → guid) | The forward map: handle id → GUID. Sized like every other per-constant table. |
| `*constant-merged-guid-table*` | `id-index` (id → list of GUIDs) | When two constants are merged (the `kb-merge` operation), the loser's GUID is preserved here under the winner's id, so dumps that reference the loser's GUID still resolve to the merged result. |
| `*constant-from-guid*` | hashtable (guid → constant) | The reverse map: GUID → constant handle. Hash test is `equalp` so two `guid` structs with the same string compare equal. |

The functions:

| Function | Purpose |
|---|---|
| `setup-constant-guid-table size exact?` | Allocate the three tables sized for the loaded KB. |
| `lookup-constant-guid id` | Forward map. |
| `lookup-constant-merged-guid id` | The merged-GUID list. |
| `lookup-constant-by-guid guid` | Reverse map. |
| `register-constant-guid id constant-guid constant` | Install in both tables. Called when a constant is created or loaded from a shell file. |
| `deregister-constant-guid id guid` | Remove from both. Called on constant deletion. |
| `clear-constant-guid-table` | Reset all three. |

`constants-high.lisp` exposes the public API in terms of "external id" — Cyc's vocabulary for "the GUID seen from outside":

```
(constant-external-id constant)        ; → (constant-guid constant)
(find-constant-by-external-id id)      ; → (find-constant-by-guid id)
(constant-external-id-p obj)           ; → (guid-p obj)
(constant-external-id-< c1 c2)         ; lexicographic on the GUID strings
(make-constant-external-id)            ; → (make-constant-guid) → (new-guid)
```

`(constant-info-from-guid-strings strs)` is the bulk lookup: take a list of GUID strings, return `(guid-string constant-name)` pairs for those that resolve.

`*constant-legacy-guid-date* '(7 20 1969)` and `constant-legacy-id-p` are the date-relative GUID hook (mostly unused; the date is the moon-landing trivia).

## When does a GUID come into being?

There are exactly **three situations** that mint or install a GUID for a constant:

1. **A user creates a fresh constant** via `(create-constant name)` or `(find-or-create-constant name)`. With no `external-id` argument supplied, `make-constant-external-id` is called, which calls `new-guid`, which generates a fresh UUID v4. The constant is registered into both directions of the map.
2. **A user creates a constant with a specific GUID** via `(create-constant name external-id)`. The two-arg form is the cross-image install path — used when constructing a constant that must match a GUID known on another image (typically the API replicates a known constant locally).
3. **The KB load reads a constant shell** off `constant-shell.cfasl` and calls `load-install-constant-ids constant dump-id guid` (`dumper.lisp`). The GUID came from the original dump; it's the same GUID that constant has had since its first creation, however many image generations ago.

There is no other path. GUIDs are not generated lazily, not derived from the constant's name, not reused — once allocated, a GUID is owned by exactly one constant for that constant's lifetime.

## When does a GUID change or disappear?

There are exactly **three situations** that mutate the table:

1. **A constant is deleted** — `deregister-constant-guid` removes both directions. The GUID is now free; it will not collide because UUID v4 is random across 128 bits, but more importantly Cyc never reuses a deleted constant's GUID for a new one (by convention, not by enforcement).
2. **Two constants are merged** — the loser's GUID is moved into `*constant-merged-guid-table*` under the winner's id. The winner's primary GUID stays the same. Both GUIDs resolve to the winner via `lookup-constant-by-guid` (since the merged map is consulted on lookup) — this is what keeps old serialized references working after a merge.
3. **The KB tables are torn down** — `clear-constant-guid-table` resets all three. Happens on full KB unload (rare; mostly during testing).

Renaming a constant does **not** touch the GUID. The GUID is identity; the name is a label.

## CFASL integration

`cfasl.lisp` defines two opcodes:

| Opcode | Wire form | Purpose |
|---|---|---|
| 25 (`*cfasl-opcode-legacy-guid*`) | UUID string (36 bytes) length-prefixed | The original encoding. Calls `cfasl-output-string` on `(guid-to-string guid)`. Decoded by `cfasl-input-legacy-guid` → `string-to-guid`. |
| 43 (`*cfasl-opcode-guid*`) | 16 raw bytes | The terse encoding, gated by `*terse-guid-serialization-enabled?*`. Calls `disassemble-guid-to-fixnums` and writes 16 bytes. The matching `cfasl-input-guid` is the TODO at `cfasl.lisp:574` — it isn't registered, so the terse-decode path doesn't currently work. |

Two further opcodes carry GUIDs as part of larger structures:
- 32 (`complete-constant`) — GUID + name string, used in `*within-complete-cfasl-objects*` mode.
- 30 (`constant`) — when in externalization mode, the body is a GUID rather than a dump-id.

GUIDs are how cross-image transport works at the CFASL level. See `cfasl.md` for the detailed dispatch.

## Dump-format presence

Every `constant-shell.cfasl` record is `(dump-id, name, guid)` (read by `load-constant-shell-internal`). The GUID is what makes this a portable shell — the dump-id is image-local but the GUID lets a different image reconstruct the same logical constant.

The GUID is **not** repeated for assertions, deductions, NARTs, KB-HL-supports, etc. Those serialize by dump-id only. They depend on the constant shells being loaded first, so the constants exist before anything tries to reference them by id (see `kb-dumper-loader.md` for the two-phase load).

## Why GUIDs aren't kept as KB facts

Constants have GUIDs but **the GUIDs aren't reflected as assertions**. There's no `(externalIDOf Foo "abc-...")` predicate in the KB. The reason is bootstrap circularity: GUIDs are needed to look up constants, so storing them as constant→GUID assertions would require a constant→GUID lookup to find the assertion that records the constant→GUID lookup. They're a runtime-only structure, populated from the constant-shell file at KB load time.

This pattern — runtime structure with no KB reflection — is rare in Cyc. It applies to GUIDs, to handle ids, and to a small number of other identity-establishing tables. Most other "metadata" about constants (`isa`, `genls`, etc.) is in the KB as ordinary assertions.

## `*unencapsulated-common-symbols*` and the encapsulation tie-in

Encapsulation (`encapsulation.lisp`) is the cross-image transport layer that uses GUIDs for constant identity. `encapsulate-constant-internal` produces `(:hp name external-id)` — the external-id slot is the constant's GUID. See `encapsulation.md` for the full transport story.

## Notes for a clean rewrite

- **Just use a real `uuid` library.** The string-as-internal-rep with regex-validation and per-byte format strings is what you get when SubL doesn't have one. CL has `uuid` (Quicklisp) — strip this whole file in favor of it.
- **Store GUIDs as 16-byte vectors, not strings.** The string form is a wire/UI format. The internal form should be raw bytes — comparisons are faster, equality is `equalp`, and the disassemble/assemble dance disappears.
- **Use `equalp` or a custom hash if you keep strings.** The current `*constant-from-guid*` uses `equalp` for case-insensitivity. If you migrate to byte-vectors, `equalp` still works, and case becomes a non-issue.
- **Drop the legacy 36-char string opcode.** Opcode 25 is paid for by every dump. Opcode 43 (terse 16-byte) is the right encoding; finish wiring up `cfasl-input-guid` and migrate the format. Old dumps can keep working via a one-way reader for opcode 25.
- **The merged-GUID table is right; keep it.** Cross-image references need to survive a constant merge, and an alias table is the minimum invasive way to do that. Just maybe rename it: `*constant-historical-guids*` or `*constant-deprecated-guids*` makes the intent clearer than "merged".
- **`constant-external-id` vs `constant-guid` is two names for the same thing.** Pick one. The "external-id" abstraction was meant to hide GUIDs in case the format changed; it never did, so the abstraction has only added noise.
- **The `id-index` for forward GUID lookup is overkill.** A flat array sized to constant-count works. You only ever look up by handle id, and ids are dense.
- **The GUID-to-constant hashtable should be the only structure exposed externally.** Per-id GUID lookup is mostly internal: the dumper, the externalization codec, and a few `constant-info-from-guid-strings`-style bulk operations. A clean rewrite can fold the forward direction into the constant struct itself (one extra slot) and drop `*constant-guid-table*` entirely.
- **Don't reinvent `date-relative-guid`.** That whole feature was an attempt to make build-id-derived GUIDs that sorted nicely and embedded the build date. UUID v7 (timestamp-prefixed) is the modern equivalent; if you want sortable identity-with-time, use that.
