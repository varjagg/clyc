# Special-variable-state

A **special-variable-state** is a snapshot of values for a set of dynamic (special) variables, packaged as a single first-class value. It captures `(variables, values)` pairs at a moment in time so they can be restored later via `progv`. Used by Cyc when a chunk of code wants to **save its current parameter settings, do work that mutates them, and restore the snapshot at the end** — but the set of parameters isn't known statically, so `let` won't suffice.

The implementation is `special-variable-state.lisp` (124 lines). It's a small file with one defstruct, one constructor, two accessors, two updaters, and one macro. **Only one user in the LarKC port**: `wff-macros.lisp`, which wraps it for the WFF (well-formedness) checking subsystem's per-context property storage.

## The struct

```
(defstruct (special-variable-state (:conc-name "SVS-"))
  variables    ; list of symbol names
  values)      ; parallel list of their current values
```

Two parallel lists. `(svs-variables svs)` and `(svs-values svs)` extract them. Both are mutable via the autogenerated setf accessors (`_csetf-svs-variables`, `_csetf-svs-values`).

`*dtp-special-variable-state*` = `'special-variable-state` is the type tag.

`sxhash-special-variable-state-method` provides a hash function via xor of variable-list hash and value-list hash. Used if SVS objects ever land in a hashtable. The TODO comment notes that SubL's `sxhash` is generic and this should be a method registered with the dispatch — in CL it's just a regular function, accessed via convention.

## When does an SVS come into being?

```
(new-special-variable-state special-variables)
```

Mints a fresh SVS. The `special-variables` arg is a list of symbol names. The constructor:

1. Copies the variables list into the new struct.
2. Allocates a parallel list of NILs of the same length for the values slot.
3. Calls `update-special-variable-state` to populate the values slot from the *current* dynamic bindings of each variable.

So `new-special-variable-state` is **a snapshot of the current bindings of these variables**. Subsequent changes to the dynamic bindings don't affect the SVS — it has its own copy.

## When does an SVS change?

| Trigger | Effect |
|---|---|
| `update-special-variable-state svs` | Re-reads the current dynamic bindings of each variable and writes them into the values list. Keeps the variables list. Used to refresh a stale snapshot. |
| `update-special-variable-value-list values variables` | The internal helper. Walks the two lists in parallel and writes `symbol-value` of each variable into the corresponding `values` cell. Returns `values`. |
| `special-variable-state-push svs variable value` | (LarKC-stripped) Add a new `(variable, value)` pair to the SVS, growing both lists. |
| `_csetf-svs-variables`, `_csetf-svs-values` | Direct setf, used in the manipulation routines. |

The "push" operation isn't currently exposed (its body is missing). It would let a piece of code dynamically extend the snapshot — useful when the set of relevant variables grows during a computation.

## When does an SVS disappear?

It's just a struct; it's GC'd when no live reference holds it. There's no explicit teardown.

The interesting lifecycle is the **dynamic-extent** binding via the macro:

```
(with-special-variable-state svs
  body...)
```

expands to roughly:

```
(let ((svs-var <svs>))
  (check-type svs-var special-variable-state)
  (progv (svs-variables svs-var)
         (svs-values svs-var)
    body...))
```

`progv` is the CL primitive that dynamically binds a *runtime list of variables* to a *runtime list of values* for the dynamic extent of the body. On entry, the original bindings are saved; on exit (whether normal or non-local), they're restored. The SVS is **read-only** during this — `body` may mutate the dynamic bindings, but those mutations are confined to the `progv` extent.

This is how a chunk of code can:

1. Snapshot its caller's settings (`new-special-variable-state`).
2. Reset / reinitialise the variables to canonical defaults.
3. Run its work.
4. Restore the caller's settings (`with-special-variable-state`).

without knowing at compile time what the relevant variables are.

The `with-special-variable-state-variables` and `with-special-variable-state-values` functions are macro helpers — they exist purely to be called inside the macro's expansion, hence the matching names. The `register-macro-helper` calls at the bottom announce them.

## How wff-macros consumes SVS

The only user. `wff-macros.lisp` defines:

```
(defmacro with-wff-properties properties &body body)
```

which creates a WFF-properties SVS, captures the current values of the WFF-related parameters specified by `properties`, and runs `body` with those snapshots restored. The wrapper:

```
(defun new-wff-special-variable-state (properties)
  (let ((svs (new-special-variable-state nil)))
    ;; populate svs with WFF-relevant variables based on properties
    ...
    svs))

(defmacro with-wff-special-variable-state (wff-svs &body body)
  (with-temp-vars (svs)
    `(let ((,svs (...some processing of wff-svs...)))
       (with-special-variable-state ,svs ,@body))))
```

This lets WFF-checking code save the user's WFF strictness flags, switch into a different validation profile, and restore on exit. The `properties` mechanism is a higher-level filter — only certain named WFF flags get included in the SVS, not every WFF-related variable.

## Notes for a clean rewrite

- **`progv` is the right primitive; SVS is the right wrapper.** Most languages have something analogous (Python: `contextvars.run`, Rust: `tokio::task_local`). Keep this pattern in a clean rewrite — but prefer the host's idiomatic mechanism.
- **The struct could be a simple `(list of (var . val) pairs)` alist.** Two parallel lists is the SubL style; an alist is the CL style. Lookup, update, and serialization all become trivial. Drop the defstruct.
- **The `sxhash-special-variable-state-method` is unused.** SVS is only ever in `progv`; nothing puts it in a hashtable. Drop the hash method.
- **`special-variable-state-push` should be inlined or dropped.** The single use case — wff-macros — knows its variables statically and doesn't need dynamic extension. Strip the missing-larkc stub.
- **Generalize beyond WFF.** SVS is a useful primitive that deserves more callers. The inference engine, for example, could use it to capture `*memoization-state*` + `*relevant-mt-function*` + a dozen other knobs around a sub-query without touching `with-memoization-state` and friends individually.
- **Make it composable.** Currently `with-special-variable-state svs1 (with-special-variable-state svs2 body)` works as nested progv. A clean rewrite should make composition explicit: `(merge-svs svs1 svs2)` returns a snapshot that captures both.
- **Add `restore-special-variable-state svs`** as the imperative inverse of `new-special-variable-state`. Currently you can only "snapshot now" or "restore for body extent" — there's no "go back to this snapshot now and stay there." Imperative restore would be useful for context-switching coroutines.
- **The `:conc-name "SVS-"` is fine.** Three-letter abbreviation, used internally only.
- **The check-type inside the macro is defensive.** A clean rewrite could move the check to the constructor (so an SVS is always valid by construction) and drop runtime checks at use sites.
